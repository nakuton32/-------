## プロジェクト名
「TeacherBoard」

## 概要
アルバイトスタッフ間の業務引継ぎを効率化し、過去のクレーム事例とそれに対する最適な解決策を迅速に共有・閲覧できるWebアプリケーション。

## 機能一覧
- **ユーザー認証・管理機能**: ログイン・新規登録（ロール権限：マネージャー / 一般スタッフ）
- **業務報告機能**: 投稿およびカテゴリー別閲覧
- **クレーム管理機能**: カテゴリー別管理・永久保存（スタッフ側は匿名・年月のみ表示）
- **業務データ自動リセット機能**: 視認性維持のため、業務報告は当日リセット。翌日引継ぎは48時間保持後にリセット。
- **フィードバック機能**: マネージャーによるクレームへの解決策投稿（ID指定による編集）

## 使用技術
- **バックエンド**: PHP, MySQL
- **フロントエンド**: HTML, CSS

## 開発背景と設計の意図
私が飲食店でのホール業務に従事する中で、「シフト間の引継ぎに問題がある」「クレームへの適切な対応方法がわからない」という2つの課題に直面し、本アプリの開発に至りました。

### 各機能の設計思想
- **ロール権限付きログイン・新規登録**
  アルバイト環境において、名前を偽った虚偽の報告を防ぐためにアカウント制を採用しました。マネージャーの登録負荷を軽減するため、アカウント自体は各スタッフが各自で登録し、権限を付与する設計としています。
  また、UIの視認性向上のため、新規登録をsubmitにし、CSSを使用してこだわりました。
- **カテゴリー別の業務報告・クレーム管理**
  引継ぎ時にスタッフが「今、どの情報を確認すべきか」を簡単に把握できるよう、情報をカテゴリー別に分類して表示しています。
- **クレーム情報の匿名化とプライバシー配慮**
  スタッフ側ページでは報告者名を非表示にし、日付も年月までに留めることで、個人の特定ができないようにし、報告しやすいように実装しました。
- **リセット機能**
  日々の業務データが累積することで見えにくくなるため、当日〜翌日の引継ぎに必要なデータは48時間保持した後に自動でリセットされる仕組みを導入し、その他のデータは当日でりっせとさせるようにしました。。
- **マネージャーによる解決策フィードバック**
  クレーム詳細に対し、マネージャーが最善の解決策を入力することで、スタッフが自信を持って対応できる体制を整えました。
  データ量が増加しても迷わず入力できるよう、編集したいのクレームIDを指定してそのデータだけを表示させて、フィードバックを入力しやすくするように設計しました。
